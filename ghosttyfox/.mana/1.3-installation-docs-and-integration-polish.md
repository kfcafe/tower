Steps:

1. Create `native-manifest/ghosttyfox.json` as a template/reference.
   Use:
   ```json
   {
     "name": "ghosttyfox",
     "description": "Ghosttyfox native messaging host",
     "path": "/ABSOLUTE/PATH/TO/ghosttyfox-host",
     "type": "stdio",
     "allowed_extensions": ["ghosttyfox@tower.dev"]
   }
   ```

   This file is a template only; real registration should be handled by the shell scripts.

2. Create `scripts/install.sh`.
   Responsibilities:
   - derive `PROJECT_DIR` from the script location
   - build native host in release mode
   - run `npm run build`
   - write the Firefox native messaging manifest to:
     `~/Library/Application Support/Mozilla/NativeMessagingHosts/ghosttyfox.json`
   - point `path` at the release binary:
     `native-host/target/release/ghosttyfox-host`
   - print short instructions for loading the extension from `about:debugging`

   Keep it robust and path-safe.
   Do not require sudo.

3. Create `scripts/dev.sh`.
   Responsibilities:
   - derive `PROJECT_DIR`
   - build native host in debug mode
   - run `npm run build`
   - write the same Firefox native messaging manifest, but pointing at:
     `native-host/target/debug/ghosttyfox-host`
   - run `npx web-ext run --source-dir extension`

   Keep it short and practical.

4. Replace `README.md` with a final public-facing version.
   It should include:
   - title and one-sentence pitch
   - explanation of what Ghosttyfox is
   - why it exists / use cases:
     - browser-centric workflows
     - Tree Style Tabs
     - local coding agent TUIs beside web docs and apps
   - architecture summary
   - requirements
   - install steps
   - dev steps
   - troubleshooting
   - note that the terminal is local and uses Firefox native messaging
   - note that `ghostty-web` provides the Ghostty-based terminal engine in the browser

   The README should feel like a real repo, not internal notes.
