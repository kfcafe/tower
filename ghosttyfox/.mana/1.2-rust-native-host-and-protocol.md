Required dependencies:
```toml
[package]

version = "0.1.0"

[dependencies]
portable-pty = "0.9"

anyhow = "1"
```

Protocol rules:
- Firefox native messaging frames are:
  - 4-byte little-endian payload length
  - UTF-8 JSON payload
- host should read and write exactly that format

Message contract:

Extension → Host:
```json
{"type":"input","data":"ls -la\n"}
{"type":"resize","cols":120,"rows":36}
```

Host → Extension:
```json
{"type":"output","data":"<base64-encoded bytes>"}
{"type":"exit","code":0}
{"type":"error","message":"..."}
```

Steps:

1. Create `src/protocol.rs`.
   Implement:
   ```rust
   pub fn read_message(reader: &mut impl Read) -> std::io::Result<serde_json::Value>
   pub fn write_message(writer: &mut impl Write, msg: &serde_json::Value) -> std::io::Result<()>
   ```

   Requirements:
   - read 4-byte LE length
   - read payload bytes
   - parse JSON
   - write JSON bytes with LE length prefix
   - flush on every write

   Use the canonical framing pattern.

2. Create `src/pty.rs`.
   Build a small wrapper around `portable_pty`.

   Behavior requirements:
   - use `native_pty_system()`
   - open PTY with initial size 80x24
   - spawn `$SHELL`, fallback `/bin/zsh`
   - set `TERM=xterm-256color`
   - expose a way to:
     - read PTY output
     - write PTY input
     - resize PTY
   - keep the abstraction minimal

   Suggested structure:
   ```rust
   pub struct PtySession { ... }

   impl PtySession {
       pub fn new(cols: u16, rows: u16) -> anyhow::Result<Self>;
       pub fn writer(&self) -> anyhow::Result<Box<dyn std::io::Write + Send>>;
       pub fn reader(&self) -> anyhow::Result<Box<dyn std::io::Read + Send>>;
       pub fn resize(&self, cols: u16, rows: u16) -> anyhow::Result<()>;
   }
   ```

   If `portable-pty`’s ownership model suggests a different but simpler API, prefer that.

3. Create `src/main.rs`.
   Responsibilities:
   - initialize the PTY session
   - create a PTY output reader thread
   - read incoming native-messaging frames from stdin in the main thread
   - dispatch `input` and `resize` messages
   - base64-encode PTY output and send `output` messages
   - on EOF / child termination, emit `exit`
   - on recoverable failure, emit `error`

   Important details:
   - guard stdout with `Arc<Mutex<Stdout>>` if writes can come from more than one place
   - use threads, not async
   - keep the message parsing explicit and local
   - do not make the worker invent a protocol; use the message contract above exactly

4. Handle input messages.
   - `type == "input"` → take `data` as a UTF-8 string and write its bytes directly to PTY stdin

5. Handle resize messages.
   - `type == "resize"` → parse integer `cols` / `rows`
   - clamp or reject absurd zero values sensibly
   - call PTY resize with `u16`

6. Handle PTY output.
   - read raw bytes from the PTY
   - base64 encode them
   - send JSON `{ "type": "output", "data": ... }`

7. Handle shutdown.
   - if PTY reader reaches EOF, send `{ "type": "exit", "code": 0 }`
   - if stdin closes, the process may exit cleanly
   - if a message is malformed, send `{ "type": "error", "message": ... }` rather than panicking

Coding guidance:
- prefer a small `main.rs` plus two helper modules, not a large multi-module architecture
- avoid extra generics / traits unless they clearly simplify code
- prioritize comprehensibility over elegance

In scope:
