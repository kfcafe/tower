## Task
When streaming fails mid-response (connection drop, SSE parse failure after partial data), fall back to a non-streaming request.

## Current behavior
If the SSE stream drops mid-response, Imp emits an error and the agent loop sees it as a failure. The partial data is lost.

## Desired behavior
Claude Code detects streaming failures and retries the exact same request with `stream: false`. The non-streaming response is then converted into the same StreamEvent sequence.

## Implementation

1. In `crates/imp-llm/src/providers/anthropic.rs`, add a non-streaming response type:
```rust
#[derive(Debug, Deserialize)]
struct ApiResponse {
    model: String,
    content: Vec<ApiResponseBlock>,
    stop_reason: Option<String>,
    usage: SseUsage,
}

#[derive(Debug, Deserialize)]
#[serde(tag = "type")]
enum ApiResponseBlock {
    #[serde(rename = "text")]
    Text { text: String },
    #[serde(rename = "thinking")]
    Thinking { thinking: String },
    #[serde(rename = "tool_use")]
    ToolUse { id: String, name: String, input: serde_json::Value },
}
```

2. Add `fn non_streaming_request(client, api_key, request) -> Result<AssistantMessage>` that sends `stream: false` and parses the JSON response directly.

3. In `stream_response()`, track whether we received partial content. If the stream errors after receiving `message_start` but before `message_stop`, retry with non-streaming.

4. Convert the `ApiResponse` into `Vec<StreamEvent>` using a helper `fn response_to_events(resp: ApiResponse) -> Vec<StreamEvent>`.

5. Non-streaming timeout: 300 seconds (Claude Code's default).

## Tests

- `test_response_to_events_conversion` — verify correct StreamEvent sequence from non-streaming response
