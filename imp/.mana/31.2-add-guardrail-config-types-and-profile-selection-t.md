Implement the configuration model for engineering guardrails in imp-core.

Files to modify:
- `crates/imp-core/src/lib.rs`
- `crates/imp-core/src/config.rs`
- `crates/imp-core/src/guardrails.rs` (new)
- `crates/imp-core/Cargo.toml` only if a small dependency wiring tweak is truly required; prefer using what is already there.

Requirements:
1. Add a new module `guardrails.rs` instead of making `config.rs` larger. `config.rs` is already large; keep new concepts isolated.
2. Define the core types in `guardrails.rs` and wire them into `Config`:
   - `GuardrailConfig`
   - `GuardrailLevel` with at least `Advisory` and `Enforce`
   - `GuardrailProfile` with at least `Auto`, `Generic`, `Zig`, `Rust`, `TypeScript`, `C`, `Go`, `Elixir`
3. `GuardrailConfig` should be serde-deserializable from TOML and support practical fields like:
   - `enabled`
   - `level`
   - `profile`
   - `critical_paths`
   - `after_write` (list of commands)
   - any small helper fields needed to support runtime integration cleanly
4. Add a helper on the config/module side to resolve the effective profile. If `profile = auto`, use the existing `project-detect` dependency to infer the project type instead of hand-rolled filename checks. Keep the profile model broad enough to support multiple built-in language profiles cleanly, including Go and Elixir, and do not add a runtime dependency on uu for this.
5. Keep this unit focused on config and profile resolution only. Do not add prompt assembly or hook execution here.
6. Add focused tests for TOML deserialization and effective-profile resolution.
7. Preserve existing config layering behavior.
8. Keep guardrails orthogonal to existing agent modes. This unit should add guardrail configuration, not introduce a new safety-specific agent mode.

Acceptance criteria:
- imp-core builds cleanly
- config can parse a `[guardrails]` table
- `Auto` can resolve to a concrete profile using project-detect
- no prompt/hook behavior added yet

Do not update README/docs in this unit; that comes later.
