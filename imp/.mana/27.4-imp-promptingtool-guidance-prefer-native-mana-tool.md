## Current State
imp already injects some guidance telling the model to prefer the native `mana` tool over `bash "mana ..."`, but this behavior should be tightened and protected by regression coverage. The goal is not a big policy engine; it is to make the native path the obvious default when an equivalent action exists.

## Task
Strengthen and verify the preference for native mana tooling.

Do the following:
1. tighten the native `mana` tool description so its orchestration capabilities are obvious
2. ensure the system-prompt/tool guidance clearly prefers native `mana` actions over `bash` for equivalent mana operations
3. if needed, add a **narrow** low-noise guardrail for obvious `bash` calls that begin with `mana ` when a native action exists
4. add regression coverage so future prompt/tool-surface changes do not silently lose this guidance

## Files to Modify
- `crates/imp-core/src/tools/mana.rs`
- `crates/imp-core/src/system_prompt.rs`
- `crates/imp-core/src/hooks.rs` — only if the optional narrow guardrail truly belongs here

## Scope Boundaries
- Do **not** add a broad shell policing system
- Do **not** block bash when there is no native mana equivalent
- Keep any guardrail low-noise and obvious to reason about

## Edge Cases
- guidance must still allow `bash` for mana-adjacent commands with no native action
- the system prompt should remain concise
- if a guardrail is added, it should only trigger on highly obvious `mana ...` shell commands
