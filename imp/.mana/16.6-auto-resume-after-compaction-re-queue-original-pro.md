When context compaction fires mid-task, the agent loses its thread. The original prompt should be re-queued with a context note so the agent resumes seamlessly.

## What to implement

1. **Track the original prompt** in the Agent struct:
   - Add `original_prompt: Option<String>` field to Agent
   - Set it at the start of `run()` before the main loop

2. **After compaction succeeds** in the agent loop (where `replace_with_compacted_messages` is called), inject a user message:
   ```
   [Context was compacted to manage the conversation length. The original request was: "{original_prompt}". Continue where you left off — check what's already been done and proceed with remaining work.]
   ```
   - This goes after the compaction summary message (which replaces old messages)
   - It becomes the next "user turn" that the model sees

3. **Don't re-queue if the agent just started** (turn 0) — compaction at turn 0 means the task is just too big, not that we need to resume.

## Key files
- `imp-core/src/agent.rs` — Agent struct (add original_prompt field), run() method (set it), compaction block (inject resume message)

## Tests
- Agent test with mock provider: trigger compaction mid-run, verify the resume message is injected
- Verify original_prompt is preserved verbatim in the resume message
- Verify no resume message on turn 0
