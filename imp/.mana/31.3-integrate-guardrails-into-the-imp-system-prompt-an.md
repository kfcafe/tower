Use the new guardrail config to affect agent behavior at runtime.

Files to modify:
- `crates/imp-core/src/system_prompt.rs`
- `crates/imp-core/src/hooks.rs`
- `crates/imp-core/src/builder.rs`
- `crates/imp-core/src/agent.rs`
- add a small helper module under `crates/imp-core/src/` if needed to keep files focused

Requirements:
1. In `system_prompt.rs`, add a dedicated guardrails prompt layer that is included only when guardrails are enabled.
2. The prompt layer must be concise and practical. It should translate the advice into agent-operational rules, not quote the paper. Emphasize mechanically checkable outcomes over stylistic doctrine. Example themes:
   - keep control flow straightforward
   - keep loops/retries/timeouts bounded
   - make error handling explicit
   - prefer small, focused changes
   - leave code warning-free and easy to verify
3. The prompt text should vary by effective profile where helpful. Zig should get the richest initial wording, while Rust, TypeScript, C, Go, and Elixir should also get sensible starter wording. Generic should stay language-neutral.
4. In runtime integration, run configured `after_write` checks when modified files match configured guardrail paths.
5. Advisory level behavior:
   - run the checks
   - surface failures clearly to the agent/user
   - do not silently ignore them
6. Enforce level behavior:
   - failed guardrail checks must block forward progress for that write/turn in a way the agent can see and fix
   - do not continue as if the write succeeded cleanly
7. Keep the implementation generic and config-driven. Do not hardcode Zig commands in the hook runner itself.
8. Keep guardrails orthogonal to existing agent modes. Do not introduce a separate safety mode in this unit.
9. Add or update tests for prompt assembly and guardrail check behavior.

Acceptance criteria:
- guardrails are injected into the system prompt only when enabled
- after-write checks run through the runtime path
- advisory and enforce differ meaningfully
- imp-core still compiles and targeted tests pass

Do not update user-facing docs in this unit.
