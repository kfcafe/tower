## Task
When the model hits `max_tokens` stop reason, Imp should transparently retry with a higher max_tokens instead of returning a truncated response.

## Current behavior
Imp defaults max_tokens to `min(model_max, 8192)`. When a response is truncated (stop_reason=max_tokens), the truncated response is returned as-is. The agent loop doesn't retry.

## Desired behavior
Claude Code's approach:
1. First request: capped at 8,000 tokens (Imp already does ~8,192)
2. If stop_reason is `max_tokens`: retry the same request with 64,000 tokens
3. Limit recovery attempts to 3

## Implementation

This belongs in the **agent loop** (imp-core), not the provider. The provider should surface the stop reason, and the caller decides to retry.

However, we CAN add a helper constant and a detection function in imp-llm:

1. In `crates/imp-llm/src/providers/anthropic.rs`, add constants:
```rust
pub const DEFAULT_MAX_TOKENS: u32 = 8_192;
pub const ESCALATED_MAX_TOKENS: u32 = 64_000;
```

2. Export these from `crates/imp-llm/src/lib.rs`

3. In `crates/imp-core/src/agent.rs` (or wherever the agent loop lives), after receiving a response:
   - Check if `stop_reason == StopReason::MaxTokens`
   - If so, and if the request used the default max_tokens, retry with `ESCALATED_MAX_TOKENS`
   - Track escalation count, cap at 3

4. The `StopReason::MaxTokens` variant already exists in Imp.

## Files
- `crates/imp-llm/src/providers/anthropic.rs` (modify — add constants)
- `crates/imp-llm/src/lib.rs` (modify — export constants)
- `crates/imp-core/src/agent.rs` (modify — add escalation logic in agent loop)

## Tests
- `test_max_tokens_escalation_constants` — verify DEFAULT and ESCALATED values
- Integration test or unit test in imp-core: mock a max_tokens response, verify retry with escalated value

## Don't
- Don't make this provider-specific — the escalation logic should work for any provider
- Don't retry infinitely — cap at 3 attempts
