Make usage persistence consistent across all imp execution paths.

Problem:
Today usage survives inconsistently depending on whether the request flowed through TUI, print mode with tools, no-tools mode, headless runs, or ImpSession. V1 reporting must stop depending on incidental behavior.

Do the following:
- Audit and fix CLI print mode with tools.
- Audit and fix CLI print mode with --no-tools.
- Audit and fix headless/unit execution paths.
- Audit and fix ImpSession-driven flows.
- Ensure TUI/session flows continue to persist data correctly using the new canonical helper path rather than bespoke per-frontend logic where practical.

Constraints:
- Keep persistence ownership in imp-core.
- Do not regress existing assistant/tool-result session history.
- Ensure in-memory/no-session runs behave sensibly: reporting should only see persisted data, but runtime should not crash when persistence is unavailable.
- Prefer a shared helper/pattern over one-off patches.

Expected outcome:
Any completed request that imp can account for should be persisted canonically when a real session exists, regardless of entry point.
