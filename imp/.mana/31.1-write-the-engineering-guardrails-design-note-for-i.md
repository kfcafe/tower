Create `ENGINEERING_GUARDRAILS.md` at the imp project root.

Purpose: lock the naming, scope, config surface, and rollout before code changes. This document is the source of truth for the implementation units that follow.

Required content:
1. Explain the feature as "engineering guardrails" for imp. Do not frame it as a branded "Power of Ten" feature. You may cite Gerard Holzmann and Stripe Minion as inspirations in a short background section, but the product language should stay neutral and practical. Make it explicit that this is for improving ordinary agent-written code quality and verification, not for introducing a special safety-critical mode.
2. State the goals: make agent-produced code easier to reason about and verify; bias toward bounded execution, explicit error handling, smaller units, zero warnings, and language-aware checks. Emphasize a short, memorable, mechanically checkable set of guardrails over a large catalogue of subjective rules.
3. Define the user-facing config model at a high level:
   - enabled on/off
   - level: `advisory` vs `enforce`
   - profile: `auto`, `generic`, `zig`, `rust`, `typescript`, `c`, `go`, `elixir`
   - critical path globs
   - after-write checks / verify commands
4. Define what happens at each level:
   - advisory: inject guidance and surface check failures clearly
   - enforce: treat configured violations/check failures as blocking
5. Make it explicit that guardrails are a project-agnostic imp feature with multiple built-in language profiles, not a Zig-only feature.
6. Include sections for the initial built-in language profiles:
   - Zig with the initial guidance and default checks:
     - `zig fmt --check .`
     - `zig build`
     - `zig build test`
     - Zig-specific advice: allocator discipline, avoid casual `catch unreachable`, explicit error handling, bounded retries/loops, small readable functions
   - Rust starter guidance/check expectations
   - TypeScript starter guidance/check expectations
   - C starter guidance/check expectations
   - Go starter guidance/check expectations
   - Elixir starter guidance/check expectations
7. Include a section on `profile = auto` and explicitly say it should use the existing `project-detect` crate rather than custom file sniffing.
8. Include a short section on boundaries: imp owns agent-time guardrails, while mana verify gates remain the completion-time proof of correctness. Mention that uu-style project commands can inform starter defaults, but imp should not depend on uu runtime behavior.
9. Include a brief non-goals section that rules out a giant policy engine, safety-critical branding, and trying to prove full correctness through prompt instructions alone.
10. Include a phased rollout: config first, prompt/hook integration second, starter profiles/docs third.
11. Include at least one concrete `.imp/config.toml` example for a Zig repo and mention how the same feature applies to Rust, TypeScript, C, Go, and Elixir.

Keep this doc implementation-oriented. No vague brainstorming. It should be short enough for engineers to use, but concrete enough that a non-thinking worker could implement from it.
