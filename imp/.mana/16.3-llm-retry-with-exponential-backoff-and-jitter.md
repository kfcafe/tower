Currently a single LLM stream error kills the turn. The spec calls for exponential backoff with jitter, Retry-After header respect, and configurable retry conditions.

## What to implement

1. **Add `RetryPolicy` struct** and `stream_with_retry` wrapper in `imp-core/src/agent.rs` (or a new `imp-core/src/retry.rs`):
   ```rust
   pub struct RetryPolicy {
       pub max_retries: u32,        // default: 3
       pub base_delay: Duration,    // default: 1s
       pub max_delay: Duration,     // default: 30s
   }
   ```
   - Retryable conditions: HTTP 429 (rate limit), 5xx (server error), timeout, connection reset
   - Non-retryable: 400, 401, 403 — fail immediately
   - Exponential backoff: delay = base_delay * 2^attempt + random jitter (0-50% of delay)
   - If Retry-After header present and <= max_delay, use it. If > max_delay, fail immediately.

2. **Wire into agent loop**: wrap the `self.model.provider.stream(...)` call with retry logic. On retryable error, wait and retry. The consumer (TUI) never sees retries — they're transparent.

3. **Add `RetryPolicy` to Agent struct** with a default. AgentBuilder can configure it from config.

4. **Detect retryable errors**: The provider's `StreamEvent::Error` needs to carry enough info (HTTP status code or error category) to distinguish retryable from non-retryable. Check what `imp_llm::Error` variants exist and whether they expose status codes.

## Key files
- `imp-core/src/agent.rs` — agent loop where stream is created, Agent struct
- `imp-llm/src/error.rs` — Error types, check what info is available
- `imp-llm/src/providers/anthropic.rs` — how errors are produced during streaming

## Tests
- Mock provider that fails N times then succeeds — verify retry works
- Mock provider that always fails — verify max_retries is respected
- Test that 401 errors are NOT retried
