## Task
Improve Imp's retry logic for the Anthropic provider to match Claude Code's resilience patterns.

## Current state
Imp retries 3 times with exponential backoff (1s, 2s, 4s) on status codes 429, 500, 502, 503, 529. It ignores the `retry-after` header and doesn't handle context overflow errors.

## Changes needed

### 1. Honor `retry-after` header
When the API returns a `retry-after` header (seconds), use that as the delay instead of exponential backoff:
```rust
fn retry_delay_with_header(attempt: u32, response: &reqwest::Response) -> Duration {
    if let Some(retry_after) = response.headers().get("retry-after") {
        if let Ok(secs) = retry_after.to_str().unwrap_or("").parse::<u64>() {
            return Duration::from_secs(secs);
        }
    }
    retry_delay(attempt) // existing fallback
}
```

### 2. Increase max retries to 8
Claude Code uses 10 retries. Bump from 3 to 8 as a reasonable middle ground.

### 3. Context overflow auto-recovery
Parse "input length and `max_tokens` exceed context limit" errors (HTTP 400):
```
input length and `max_tokens` exceed context limit: 188059 + 20000 > 200000
```
Extract input_tokens, max_tokens, context_limit. Calculate `available = context_limit - input_tokens - 1000` (safety buffer). Retry with reduced max_tokens. Floor at 3000 tokens.

### 4. Track consecutive 529 errors
After 3 consecutive 529 (overloaded) errors, emit a specific error variant instead of retrying forever. This prevents infinite loops during capacity cascades.

## Implementation

1. Refactor `stream_response()` to capture the response status and headers before deciding whether to retry
2. Add `parse_context_overflow(body: &str) -> Option<(u32, u32, u32)>` that extracts (input_tokens, max_tokens, context_limit)
3. Add `consecutive_529: u32` counter in the retry loop
4. Change `MAX_RETRIES` from 3 to 8
5. When retrying with reduced max_tokens, rebuild the ApiRequest with the new value
