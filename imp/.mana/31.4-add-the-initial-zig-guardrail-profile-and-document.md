Add the initial built-in guardrail profile set and document guardrails as a project-agnostic imp feature.

Files to modify:
- `crates/imp-core/src/guardrails.rs`
- `README.md`
- `ENGINEERING_GUARDRAILS.md` only if the implementation requires a small correction to the earlier design note
- an example config file or README config block if helpful

Requirements:
1. Keep the built-in guardrail architecture profile-driven and project-agnostic. Do not let the docs or code imply that guardrails are a Zig-only feature.
2. Ship more than one built-in language profile. At minimum, add built-in profiles for Zig, Rust, TypeScript, C, Go, and Elixir.
3. Zig should remain the most polished initial built-in profile. Add concrete Zig defaults to the built-in profile behavior:
   - `zig fmt --check .`
   - `zig build`
   - `zig build test`
4. Add sensible starter defaults and guidance for Rust, TypeScript, C, Go, and Elixir. These do not need the same level of polish as Zig yet, but they must be real built-in profiles rather than placeholders. Treat them as practical starter profiles for improving agent-written code quality and verification, not as exhaustive safety packs.
5. Zig-specific guidance should cover the practical issues we care about in agent-written Zig code:
   - explicit error handling
   - allocator ownership/lifetime discipline
   - avoid casual `catch unreachable`
   - bounded loops/retries/buffers where relevant
   - small readable functions and minimal hidden control flow
6. If `profile = auto` detects one of the supported languages through `project-detect`, the effective behavior should pick that profile automatically.
7. Update `README.md` so a user switching to imp can discover that guardrails are a general imp feature. Document:
   - that guardrails are for helping agents write better and safer code in ordinary projects
   - what guardrails are
   - the config block
   - advisory vs enforce
   - `profile = "auto"`
   - how language profiles fit in
   - the initial built-in profiles (Zig, Rust, TypeScript, C, Go, Elixir)
   - Zig as one concrete example, not the whole feature
8. Keep the docs honest about what is built in versus configurable.
9. Add focused tests for built-in profile defaults if the code structure supports it.

Acceptance criteria:
- guardrails are described and documented as project-agnostic
- built-in profiles for Zig, Rust, TypeScript, C, Go, and Elixir exist in code
- README documents the general feature and includes at least one concrete example
- `profile = auto` can select a supported built-in profile when project-detect reports it
- imp-core builds and targeted tests pass

Do not add support for every language under the sun in this unit. The architecture must stay general, but the initial built-in profile set should be useful on day one. Use uu-style project command expectations only as design inspiration where helpful; do not make imp depend on uu runtime behavior in this unit.
