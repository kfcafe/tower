Goal: implement soul.md as the canonical personality source in imp, replacing the current config-backed personality flow over time.
Context: Planning is complete in docs/proposals/soul-md-design-2026-04-05.md and mana epic .7. The implementation should follow the agreed model: soul.md is the source of truth, one tunable equals one stable line in a builder-managed `## Tunables` section, the rest of the markdown remains freeform, exact-match preset detection only, and edited lines require diff-confirm overwrite from the builder.

Implementation stages:
1. Add soul.md loading and prompt integration while preserving fallback to current config personality.
2. Add soul.md parsing/matching utilities for the builder-managed `## Tunables` section.
3. Update the /personality TUI flow to use soul.md as source of truth with Builder and Source tabs, including edited/mod state and diff-confirm overwrite behavior.
4. Seed/generate default soul.md content from current personality defaults and/or migrate existing config personality into a first soul.md.
5. Deprecate old config-backed personality storage/path where safe.
6. Verify, commit, and build release.

In scope:
- imp-core personality + prompt integration
- imp-tui personality UX
- soul.md document handling
- migration path
- verification and release build

Out of scope:
- wizard personality UI unless needed incidentally
- speculative reset/recovery systems
- fuzzy matching or hidden metadata

Hard constraints:
- no frontmatter schema as the core model
- one tunable = one stable line
- source document is canonical
- builder is just a structured editor over the source
- edited lines must not be silently overwritten
