Add first-class automatic context compaction to imp on top of the newly implemented manual `/compact` system.

Why:
- Manual `/compact` now exists and is functional, but long-running sessions still rely on the user to trigger it.
- `imp-core` now has the necessary foundations: branch-local compaction semantics, deterministic pre-summary reduction, a local structured compaction executor, and context assembly that respects compacted active history.
- Research from Codex, Claude Code, Hermes, and public agent compaction systems shows the next major quality step is layered automatic compaction near context limits: prune/mask old observations first, then compact older history automatically before the model hits failure or degraded behavior.

Current review findings from the manual `/compact` implementation that should shape this epic:
1. TUI manual compaction currently duplicates provider/model/auth setup logic that also exists in normal prompt/session startup paths.
2. Manual compaction currently executes synchronously in the UI path; auto-compaction must be non-blocking and robust around active agent/task lifecycle.
3. TUI compaction rendering currently infers compaction display items via the summary prefix string. Auto-compaction should introduce more explicit metadata/representation where practical.
4. Provider-native compaction has a typed seam (`CompactionStrategy`) but is intentionally not enabled yet.

Target outcome:
- imp automatically compacts older context when the active context approaches a configurable threshold, without requiring the user to type `/compact`.
- Auto-compaction preserves recent working turns verbatim, summarizes older history, and keeps future turns using the compacted active branch view.
- Auto-compaction is non-blocking, safe, observable, and configurable.

Functional goals:
1. Policy + thresholds
   - Define config for auto-compaction policy in `imp-core` config.
   - Support at least:
     - disabled/manual only
     - near-threshold automatic compaction
     - optional aggressive mode for very long sessions
   - Expose thresholds in terms of context-window ratio and/or token budget margins.
   - Integrate with existing context token estimation/reporting paths.

2. Triggering semantics
   - Define when auto-compaction runs:
     - pre-turn before sending a request if active context is near threshold
     - possibly post-turn or idle-time follow-up later, but pre-turn should come first
   - Ensure it does not trigger while an agent turn is actively streaming.
   - Prevent compaction loops or repeated compaction every turn via cooldown/circuit-breaker logic.
   - Define behavior when the session is already recently compacted or history is too short.

3. Shared compaction execution path
   - Refactor current manual compaction plumbing so manual and automatic compaction share the same core execution path and provider/model/auth setup helper(s).
   - Avoid duplicating summarizer creation logic in the TUI.
   - Make the shared path reusable from both CLI/core session logic and TUI runtime orchestration.

4. Non-blocking runtime integration
   - Run auto-compaction in a way that does not freeze the TUI event loop.
   - Integrate with existing agent task/runtime signal handling so auto-compaction can happen safely before a turn or as a managed background preparation step.
   - Ensure agent/task replacement semantics remain correct if auto-compaction occurs around queued follow-up turns.

5. UX + observability
   - Surface when auto-compaction happened and why (e.g. context nearing limit).
   - Distinguish auto-compaction from manual `/compact` in user-visible status/transcript/session metadata.
   - Improve compaction display semantics so the UI does not rely only on string-prefix heuristics where a more explicit representation is available.
   - Keep reload/resume/tree behavior coherent.

6. Safety + recovery
   - Reuse deterministic shrinking and retry behavior from manual compaction.
   - Add circuit breakers for repeated auto-compaction failures.
   - Define fallback when summarization/auth/provider setup is unavailable during an auto-compaction opportunity:
     - continue without compaction and warn, or
     - rely on masking/truncation first, depending on policy.
   - Ensure tool-call/result invariants and preserved-tail guarantees remain intact.

7. Future-facing provider seam
   - Keep auto-compaction architecture compatible with the existing `CompactionStrategy` seam.
   - Do not make provider-native compaction mandatory, but allow future support without redesigning policy/trigger logic.

Suggested decomposition:
A. Auto-compaction config + policy design in `imp-core`.
B. Shared compaction runner refactor (manual + auto use same execution path).
C. Pre-turn threshold detection and trigger plumbing.
D. Non-blocking TUI/runtime integration and user-visible status.
E. Explicit compaction UI/session metadata improvements.
F. Tests for threshold triggering, cooldown/circuit-breaker behavior, and follow-up turn correctness.
G. Optional provider-native auto-compaction enablement behind capabilities/config.

Acceptance criteria:
- imp can automatically compact older history near the context limit without a manual slash command.
- Auto-compaction uses the same safe compaction semantics as manual compaction.
- The implementation is non-blocking and does not regress agent/task lifecycle handling.
- Users can tell when auto-compaction occurred.
- Auto-compaction is configurable and can be disabled.
- Repeated failures do not cause compaction loops or unusable sessions.
- Future provider-native compaction remains an optional optimization, not the canonical path.
