The heart of imp. The ReAct reasoning cycle that connects the LLM to tools.

## What to implement

### File: `crates/imp-core/src/agent.rs`

**Agent struct:**
```rust
pub struct Agent {
    pub model: Model,
    pub thinking_level: ThinkingLevel,
    pub tools: ToolRegistry,
    pub messages: Vec<Message>,
    pub system_prompt: String,
    pub cwd: PathBuf,
    pub max_turns: u32,  // default: 50
    pub ui: Arc<dyn UserInterface>,

    event_tx: mpsc::UnboundedSender<AgentEvent>,
    command_rx: mpsc::UnboundedReceiver<AgentCommand>,
}
```

**AgentCommand enum:** Cancel, Steer(String), FollowUp(String)

**AgentEvent enum:** (see imp_core_plan.md §2 for the full list — AgentStart, AgentEnd, TurnStart, TurnEnd, MessageStart, MessageDelta, MessageEnd, ToolExecutionStart, ToolExecutionUpdate, ToolExecutionEnd, CompactionStart, CompactionEnd, Error)

**Agent::run(&mut self, prompt: String) -> Result<()>**

The core loop:
1. Push user message onto self.messages
2. Loop until max_turns:
   a. Check command_rx for Cancel/Steer (non-blocking poll)
   b. Emit TurnStart
   c. Build Context from self.messages + self.system_prompt
   d. Build RequestOptions from model, thinking_level, tool definitions
   e. Call provider.stream() and consume the event stream
   f. For each StreamEvent, emit corresponding AgentEvent (MessageDelta for text/thinking, ToolExecutionStart for tool calls)
   g. Accumulate the full AssistantMessage from stream events
   h. Push AssistantMessage onto self.messages
   i. Extract tool calls from the assistant message
   j. If no tool calls → emit TurnEnd, break (model is done)
   k. Execute tools (concurrent reads, sequential writes — see below)
   l. For each tool result, push ToolResultMessage onto self.messages
   m. Emit TurnEnd
3. Emit AgentEnd with accumulated usage/cost

**Tool execution strategy:**
- Partition tool calls into readonly and mutable
- Execute all readonly concurrently with `tokio::JoinSet` or `FuturesOrdered`
- Then execute mutable tools sequentially, checking for Cancel between each
- Before each tool: validate params against JSON schema (use `jsonschema` crate)
- Wrap each execution with timeout (default: 120s per tool)
- If validation fails, return error as tool result (let model self-correct)

**Streaming consumption:**
- Consume the `Pin<Box<dyn Stream<Item = Result<StreamEvent>>>>` from the provider
- Accumulate text deltas into the full assistant message
- Accumulate tool call deltas (Anthropic sends tool args as JSON chunks)
- Forward deltas as AgentEvents for real-time display

**Agent::new() builder pattern:**
```rust
Agent::builder()
    .model(model)
    .thinking_level(ThinkingLevel::Medium)
    .tools(registry)
    .system_prompt("You are imp...")
    .cwd(PathBuf::from("."))
    .max_turns(50)
    .ui(Arc::new(NullInterface))
    .build() -> (Agent, mpsc::UnboundedReceiver<AgentEvent>, mpsc::UnboundedSender<AgentCommand>)
```

The caller gets the event receiver and command sender. This is how the TUI (or RPC, or headless mode) communicates with the agent.

## Acceptance criteria
- `cargo test -p imp-core -- agent` passes
- Test with a MockProvider that returns canned responses:
  - Test 1: Simple prompt → text response → agent stops (no tool calls)
  - Test 2: Prompt → tool call → tool result → text response → agent stops
  - Test 3: Prompt → multiple tool calls → results → follow-up tool calls → done
  - Test 4: Cancel command mid-run → agent stops
  - Test 5: Max turns exceeded → agent stops with error event
  - Test 6: Tool validation failure → error tool result → model self-corrects
- Verify events are emitted in correct order (AgentStart → TurnStart → ... → TurnEnd → AgentEnd)
- Verify accumulated usage/cost is correct across turns
