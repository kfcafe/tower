Create the canonical usage-accounting design and core data model for imp.

Scope:
- Audit the existing usage-bearing paths in imp-llm, imp-core, imp-cli, imp-tui, and session persistence.
- Add a versioned usage record type suitable for session custom entries. It must capture enough information for reliable historical reporting.
- Define the dedupe key/semantics needed for forked or copied session history so a request is only counted once in global summaries.
- Provide helpers in imp-core for reading usage from both canonical custom entries and legacy assistant-message usage.

The canonical usage entry should include at least:
- stable request/record id
- timestamp
- provider
- model
- assistant/session linkage needed for dedupe and attribution
- turn index or similar local attribution
- raw usage fields: input/output/cache read/cache write
- stored cost breakdown and total cost at request time
- provenance/source marker for canonical vs legacy-derived records

Design rules:
- Use SessionEntry::Custom rather than inventing a second canonical store.
- Preserve backward compatibility with existing sessions by supporting assistant-message fallback in the read path.
- Do not yet build the full CLI UX in this unit.
- Keep the schema small and versioned; do not create a giant analytics/event taxonomy.

Primary files likely involved:
- crates/imp-core/src/session.rs
- crates/imp-core/src/lib.rs
- crates/imp-llm/src/usage.rs
- any new imp-core module dedicated to usage/session accounting

Deliverable:
- core types/helpers land with tests, and there is one obvious canonical schema for later units to persist and query.
