Lua 5.4 extension runtime via mlua. Extensions can register tools, hook events, interact with the UI, and manage state.

## Crate: `crates/imp-lua/`

### Dependencies
- mlua = { version = "0.10", features = ["lua54", "async", "send", "serialize"] }
- imp-core (path)
- imp-llm (path)
- serde_json, tokio

### File: `src/loader.rs` — Extension discovery + loading

**Discovery paths:**
- ~/.config/imp/lua/*.lua (user global)
- ~/.config/imp/lua/*/init.lua (user global, directory)
- .imp/lua/*.lua (project local)
- .imp/lua/*/init.lua (project local, directory)

**Loading:**
- Create a Lua state per session (not per extension — all extensions share one state)
- Register host API functions into the `imp` global table
- Execute each extension file — the file's top-level code runs, registering hooks and tools
- Track loaded extensions for hot reload

**Hot reload (`/reload`):**
1. Fire on_session_shutdown for current state
2. Drop the Lua state entirely (frees all Lua memory)
3. Create new Lua state, re-register host API
4. Re-discover and load extensions
5. Fire on_session_start

### File: `src/bridge.rs` — Host API exposed to Lua

**imp.on(event_name, handler):**
- Register a Lua function as a hook callback
- Supported events: all HookEvent variants as strings
- The handler receives (event_table, ctx_table) where:
  - event_table: Lua table with event-specific fields
  - ctx_table: Lua table with ui methods, session access, etc.
- Handler can return a table with { block, reason, modified_content }

**imp.register_tool(definition):**
- definition: Lua table with { name, label, description, readonly, params, execute }
- params: Lua table describing JSON schema (converted to serde_json::Value)
- execute: Lua function(call_id, params, ctx) -> table with { content, details, is_error }
- Register as a LuaTool in the ToolRegistry

**imp.exec(command, args, opts):**
- Run a shell command synchronously from Lua's perspective (async under the hood via mlua)
- args: optional string array
- opts: { timeout, cwd }
- Returns: { stdout, stderr, exit_code }

**ctx.ui.* methods:**
- notify(message, level)
- confirm(title, message) → boolean
- select(title, options) → string or nil
- input(title, placeholder) → string or nil
- set_status(key, text)
- set_widget(key, lines_or_nil)
- set_footer(render_fn)
- set_working_message(text)
- custom(builder_fn) → value or nil
All backed by the UserInterface trait. Async operations use mlua's coroutine bridging.

**imp.register_command(name, definition):**
- definition: { description, handler(args, ctx) }

**imp.register_shortcut(key, definition):**
- definition: { description, handler(ctx) }

**imp.session.*:**
- get_entries() → table of session entries
- get_branch() → table of current branch entries
- append_entry(type, data) → persist extension state

**imp.send_message(message, opts) / imp.send_user_message(content, opts):**
- Inject messages into the conversation

**imp.get_model() / imp.set_model(provider, id):**
**imp.get_thinking_level() / imp.set_thinking_level(level):**
**imp.get_active_tools() / imp.set_active_tools(names):**

**imp.events.on(name, handler) / imp.events.emit(name, data):**

### File: `src/sandbox.rs` — Lua state management

**LuaRuntime struct:**
```rust
pub struct LuaRuntime {
    lua: Lua,
    tools: Vec<LuaToolHandle>,
    hooks: Vec<LuaHookHandle>,
    commands: Vec<LuaCommandHandle>,
}
```

- Manages the Lua state lifecycle
- Tracks registered tools/hooks/commands for cleanup on reload
- Provides access to registered tools for the ToolRegistry

## Acceptance criteria
- Lua extensions load from discovery paths
- imp.on() registers hooks that fire on correct events
- imp.register_tool() creates a tool callable by the agent
- imp.exec() runs shell commands and returns results
- ctx.ui.confirm() works with NullInterface (returns nil)
- Hot reload: drop state, reload, verify tools/hooks re-registered
- Error handling: Lua errors caught and logged, don't crash the runtime
- Multiple extensions load and coexist in the same Lua state
