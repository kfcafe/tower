Second and third LLM providers.

## OpenAI provider — `crates/imp-llm/src/providers/openai.rs`

**API:** OpenAI Responses API (POST https://api.openai.com/v1/responses)
- This is the newer API (not /v1/chat/completions). Use it.
- Headers: `Authorization: Bearer {key}`, `Content-Type: application/json`
- Set `stream: true`

**Message format:**
- User: `{ role: "user", content: string | ContentPart[] }`
- Assistant: `{ role: "assistant", content: string | ContentPart[] }`
- Tool results: `{ type: "function_call_output", call_id: string, output: string }`
- Tool calls appear as `{ type: "function_call", name: string, arguments: string, call_id: string }`

**Thinking/reasoning:** Map ThinkingLevel to `reasoning_effort`:
- Off: don't include reasoning parameter
- Minimal/Low: "low"
- Medium: "medium"  
- High/XHigh: "high"
- Only works with o-series models (o3, o4-mini). For non-reasoning models, ignore thinking level.

**SSE events:** OpenAI streams `response.output_item.*` and `response.content_part.*` events.
- `response.created` → MessageStart
- `response.output_item.added` → track new output item
- `response.content_part.delta` → TextDelta
- `response.output_item.done` → if function_call, emit ToolCall
- `response.completed` → MessageEnd with usage

**Caching:** OpenAI uses automatic session-based caching. No explicit cache control headers needed. Just track cache_read_tokens from usage stats.

**Image workaround:** OpenAI cannot accept images in tool results. If a tool result contains an image:
1. Replace image in tool result with text placeholder: "[Image attached below]"
2. Inject a user message after the tool result containing the image
This transform happens during message serialization in the provider module.

**Auth:** OPENAI_API_KEY env var, or stored credential.

## Google provider — `crates/imp-llm/src/providers/google.rs`

**API:** Gemini API (POST https://generativelanguage.googleapis.com/v1beta/models/{model}:streamGenerateContent?key={key})
- API key goes in URL query param, not header

**Message format:**
- `contents: [{ role: "user"|"model", parts: [...] }]`
- Text: `{ text: "..." }`
- Images: `{ inline_data: { mime_type: "image/png", data: "base64..." } }`
- Tool calls: `{ function_call: { name, args } }`
- Tool results: `{ function_response: { name, response: { content: "..." } } }`

**Thinking:** Map ThinkingLevel to `generationConfig.thinkingConfig`:
- Off: omit thinkingConfig
- Others: `{ thinkingConfig: { thinkingBudget: N } }` where N maps similarly to Anthropic

**SSE streaming quirks:**
- Google streams chunks that may contain partial tool call arguments
- Must accumulate function_call args across chunks before emitting ToolCall event
- Response uses `candidates[0].content.parts` for content blocks

**Auth:** GOOGLE_API_KEY env var, or stored credential.
