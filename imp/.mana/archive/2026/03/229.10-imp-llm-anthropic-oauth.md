### File: `crates/imp-llm/src/oauth/pkce.rs`

**PKCE generation:**
- Generate 32 random bytes, base64url-encode as verifier
- SHA-256 hash the verifier, base64url-encode as challenge
- Use `ring` or `sha2` crate for SHA-256

### OAuth login flow:

1. Generate PKCE verifier + challenge
2. Start a local HTTP server on 127.0.0.1:53692 using `hyper` or `tiny_http`
3. Build authorization URL with params: client_id, response_type=code, redirect_uri, scope, code_challenge, code_challenge_method=S256, state=verifier, code=true
4. Open the URL in the user's browser (`open` on macOS, `xdg-open` on Linux)
5. Wait for the callback OR manual code input:
   - Callback server receives code + state at /callback
   - State must match verifier
   - Return success HTML page
6. If callback server doesn't receive code (e.g., browser is on another machine):
   - Accept manual input: the user pastes the authorization code or the full redirect URL
   - Parse the code from the URL or raw input
   - For manual input, use MANUAL_REDIRECT_URI in the token exchange
7. Exchange authorization code for tokens:
   - POST to TOKEN_URL with: grant_type=authorization_code, client_id, code, state, redirect_uri, code_verifier
   - Parse response: { access_token, refresh_token, expires_in }
   - Calculate expires_at = now + expires_in - 5 minutes (buffer)
8. Store credential via AuthStore

### Token refresh:

```rust
pub async fn refresh_token(refresh_token: &str) -> Result<OAuthCredential>
```
- POST to TOKEN_URL with: grant_type=refresh_token, client_id, refresh_token, scope

### Auth storage with file locking (critical for multi-instance)

Pi uses file locking to prevent race conditions when multiple imp instances try to refresh tokens simultaneously. imp must do the same.

**File: `crates/imp-llm/src/auth.rs`** (expand the existing AuthStore):

```rust
impl AuthStore {
    /// Get API key with automatic OAuth refresh.
    /// Resolution order:
    /// 1. Runtime override (CLI --api-key)
    /// 2. API key from auth.json
    /// 3. OAuth token from auth.json (auto-refreshed with locking)
    /// 4. Environment variable (ANTHROPIC_API_KEY, etc.)
    /// 5. Fallback resolver (custom provider keys from models.toml)
    pub async fn get_api_key(&self, provider: &str) -> Option<String>

    /// Refresh OAuth token with file locking.
    /// Uses `fs2` crate for cross-platform file locking (flock on Unix).
    /// Flow:
    /// 1. Acquire exclusive lock on auth.json
    /// 2. Re-read file (another instance may have already refreshed)
    /// 3. If token still expired, call provider.refresh_token()
    /// 4. Write updated credentials
    /// 5. Release lock
    /// If lock acquisition fails after retries, read file anyway
    /// (another instance likely succeeded).
    async fn refresh_with_lock(&self, provider: &str) -> Result<OAuthCredential>
}
```

**auth.json format:**
```json
{
  "anthropic": {
    "type": "oauth",
    "access": "sk-ant-...",
    "refresh": "rt-...",
    "expires": 1710000000000
  },
  "openai": {
    "type": "api_key",
    "key": "sk-..."
  }
}
```

File permissions: 0o600 (owner read/write only). Parent directory: 0o700.

**`fs2` crate** for file locking — cross-platform (flock on Unix, LockFileEx on Windows).
Lock with retry: up to 10 attempts, 100ms backoff, 30s stale lock timeout.

### TUI Login Flow

The `/login` command in interactive mode:
1. Show an OAuth provider selector (list: Anthropic, with ✓ for already logged in)
2. User selects provider → start login flow
3. Replace editor area with login dialog showing:
   - Provider name
   - Authorization URL (clickable via terminal hyperlink escape)
   - Instructions: "Complete login in your browser. If the browser is on another machine, paste the redirect URL here."
   - Text input for manual code entry
   - Progress messages ("Exchanging authorization code...")
   - Cancel with Escape
4. On success: store credential, dismiss dialog, notify "Logged in to Anthropic"
5. On failure: show error, dismiss dialog

The login dialog communicates with the OAuth flow via callbacks:
- `on_auth(url, instructions)` → display URL and open browser
- `on_manual_code_input()` → show text input, resolve when user submits
- `on_prompt(message, placeholder)` → show prompt if callback server fails
- `on_progress(message)` → show status update

### `/logout` command
- Show provider selector with only logged-in providers
- Remove credential from auth.json
- Notify "Logged out of Anthropic"
