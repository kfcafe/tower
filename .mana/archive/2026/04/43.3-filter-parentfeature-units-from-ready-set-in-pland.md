## Problem

`plan_dispatch` in `mana/crates/mana-cli/src/commands/run/plan.rs` (lines 83-173) filters candidate units by `has_verify && status == Open`, but does not exclude feature/parent units that have open children. When a parent feature unit has `has_verify: true` AND has open child units, it gets surfaced as `ready` alongside its children. This clutters the ready set and leads to the parent being dispatched for execution, which is almost never the intent.

Similarly, `ready_units` in `mana/crates/mana-core/src/api/mod.rs` and the inline `next` handler in `imp/crates/imp-core/src/tools/mana.rs` filter for `has_verify && status == Open` but don't filter out parents with open children.

## Solution

### 1. `plan_dispatch` — already partially handled

The function already calls `has_open_descendants()` when a filter_id is provided (line 114-119). Extend this filtering to the non-filtered case:

In `mana/crates/mana-cli/src/commands/run/plan.rs`, after the `candidate_entries` filter (around line 107), add:

```rust
// Exclude units that have open descendant units — they're parent/feature
// containers and shouldn't be dispatched while children are still pending.
candidate_entries.retain(|entry| !has_open_descendants(&index, &entry.id));
```

This goes after the initial filter and before the filter_id check.

### 2. `ready_units` in `mana-core/src/api/mod.rs` (lines 411-430)

Add the same check. The function has access to the `index`. Add a helper or inline check:

```rust
pub fn ready_units(index: &Index) -> Vec<&IndexEntry> {
    let closed_ids: std::collections::HashSet<&str> = index
        .units
        .iter()
        .filter(|e| e.status == Status::Closed)
        .map(|e| e.id.as_str())
        .collect();

    index
        .units
        .iter()
        .filter(|e| {
            e.status == Status::Open
                && e.has_verify
                && e.dependencies
                    .iter()
                    .all(|dep| closed_ids.contains(dep.as_str()))
                // Don't surface parents that still have open children
                && !has_open_children(e, index)
        })
        .collect()
}

fn has_open_children(entry: &IndexEntry, index: &Index) -> bool {
    index.units.iter().any(|e| {
        e.parent.as_deref() == Some(&entry.id)
            && e.status != Status::Closed
    })
}
```

### 3. `next` handler in ManaTool (imp-core/src/tools/mana.rs)

The inline `next` implementation already filters `!e.feature`, which helps. But a non-feature parent with verify can still leak through. Add:

```rust
.filter(|e| {
    e.status == mana_core::unit::Status::Open
        && e.has_verify
        && !e.feature
        && mana_core::blocking::check_blocked(e, &index).is_none()
        // Don't recommend parents with open children
        && !index.units.iter().any(|child|
            child.parent.as_deref() == Some(e.id.as_str())
            && child.status != mana_core::unit::Status::Closed
        )
})
```

## Don't
- Don't remove the `has_verify` filter — verifyless units still shouldn't be dispatched
- Don't change the feature flag semantics
- Don't modify the existing `has_open_descendants` function
- Don't change wave computation logic
