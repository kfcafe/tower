## What to implement
Make the Tower-level config-centric rule explicit and consistent across the root documentation and agent instructions.

## Files
- `README.md`
- `VISION.md`
- `UMBRELLA.md`
- `AGENTS.md`

## Context
Tower now has a documented split between shared project state (`.mana/`) and local Wizard UI state (`.wizard/`). The missing cross-project rule is that runtime policy should be controlled through explicit configuration rather than hidden code constants or one-off flags. The root docs already describe per-app config surfaces, but this unit should make the contract crisp and consistent:
- shared behavior belongs in repo config
- personal defaults belong in user config
- secrets stay out of committed config
- env vars and CLI flags override config instead of replacing it

## Acceptance criteria
- root docs clearly describe Tower as config-centric
- `AGENTS.md` tells agents to prefer config over scattered code-level policy
- the docs preserve the distinction between `.mana/`, `.wizard/`, and config surfaces
- wording is consistent enough that later app-specific units can point back to this contract
