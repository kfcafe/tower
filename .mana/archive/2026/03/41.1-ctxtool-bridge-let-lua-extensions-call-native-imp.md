Add ctx.tool(name, params) to the Lua host API so extensions can call any native imp tool.

The existing code in bridge.rs passes ctx as a simple Lua table with just `cwd` and `cancelled`. Extend it:

1. In `setup_host_api()`, add an `imp.tool(name, params)` function
2. This function needs access to the ToolRegistry to look up tools
3. Call `tool.execute(call_id, params_json, tool_ctx)` 
4. Convert the ToolOutput back to a Lua string (text content) or table (structured)
5. Also add `imp.http.get(url, headers?)` and `imp.http.post(url, body, headers?)` using reqwest

The tricky part: tool.execute() is async, but mlua functions are sync by default. Use `mlua` async support or `tokio::runtime::Handle::current().block_on()`.

Also add `imp.env(name)` with scoping — the extension's declared `env` table limits which vars are accessible.

### Files
- `imp/crates/imp-lua/src/bridge.rs` — add imp.tool(), imp.http, imp.env()
- `imp/crates/imp-lua/src/sandbox.rs` — store ToolRegistry reference and allowed env vars

### Tests
- Test imp.tool("read", { path = "..." }) returns file content
- Test imp.http.get with httpbin.org or mock
- Test imp.env scoping (allowed vs denied)
