## What to implement
Define Familiar's configuration model while the platform is still early, so it follows the same config-centric rules as the rest of Tower instead of accreting ad-hoc runtime policy later.

## Files
- `familiar/README.md`
- `familiar/plan.md`
- optionally `familiar/ESTIMATES.md` if configuration work needs sizing notes

## Context
Familiar is still plan-heavy, which makes this the right moment to define configuration boundaries. The plan already talks about team-configurable settings such as service credentials, cost caps, permissions, and tool configuration. This unit should separate:
- deployment/runtime config
- team-managed settings stored by the app
- secrets and credentials
- per-user preferences
- cross-project assumptions inherited from Tower

## Acceptance criteria
- Familiar docs describe a concrete config philosophy and layering model
- docs distinguish app config from user/team data stored by the platform
- secrets and provider credentials are not treated as committed repo config
- later Familiar implementation work has a stable config contract to build from
