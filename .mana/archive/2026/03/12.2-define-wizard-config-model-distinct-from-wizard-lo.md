## What to implement
Define the first-class Wizard configuration model and make it distinct from `.wizard/` local view state. This unit can be documentation-first, but it should be concrete enough that later code can implement it without revisiting the storage boundaries.

## Files
- `wizard/README.md`
- `wizard/SPEC.md`
- `wizard/BACKEND_ARCHITECTURE.md`
- `wizard/FULLSTACK_ARCHITECTURE.md`
- optionally `wizard/crates/wizard-store/src/lib.rs` and/or `wizard/crates/wizard-orch/src/lib.rs` if you add initial config types or loaders

## Context
Wizard already documents `.mana/` as canonical shared project state and `.wizard/` as personal local state. That makes `.wizard/` the wrong place for shared orchestration policy. This unit should define:
- what counts as Wizard config vs `.wizard/` local state
- where user config lives (`~/.config/wizard/config.toml` is already referenced in root docs)
- whether shared project-level Wizard behavior lives in repo config, and at which path
- the override order between built-in defaults, user config, project config, env vars, and CLI commands

## Acceptance criteria
- Wizard docs describe a concrete config surface and override order
- Wizard docs keep personal layout state in `.wizard/` and do not overload it with shared policy
- if code is added, it compiles with `cargo check -p wizard-orch -p wizard-store`
- later Wizard implementation work has a stable config contract to build on
